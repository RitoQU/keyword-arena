## 一、项目概述

一款浏览器端像素风对战游戏。玩家通过输入 1-3 个关键词生成独特游戏角色，角色拥有基于 D&D 体系的多维属性、武器、防具、技能和特殊物品，可与其他玩家或系统角色进行回合制自动对战。

---

## 二、技术方案

### 2.1 推荐技术栈

| 层级 | 技术 | 选择理由 |
|------|------|----------|
| **框架** | Next.js 14 (App Router) + TypeScript | 全栈一体，Vibecoding 支持最好，AI 代码生成准确率最高 |
| **样式** | Tailwind CSS + 像素风自定义主题 | 快速开发，像素风通过字体和 CSS 实现 |
| **数据库** | Supabase (PostgreSQL) | 免费额度充足，自带 REST API，无需自己管数据库 |
| **LLM** | MiniMax API（订阅计划） | 用户已有订阅，中文能力强，兼容 OpenAI SDK 格式 |
| **部署** | Vercel | 一键部署，免费额度足够个人项目，全球 CDN |
| **版本控制** | GitHub | 与 Vercel 集成，代码管理 |

### 2.2 为什么不用 Python 后端？

你熟悉 Python 生态，但这个项目选 Next.js 的理由：
- **全栈一体**：前端页面和后端 API 在同一个项目里，不用分两个仓库
- **部署极简**：Vercel 一键部署，不需要自己配服务器
- **Vibecoding 标配**：Next.js + React 是 AI 辅助编程支持最好的技术栈，Copilot 生成代码的准确率最高
- **学习价值**：JavaScript/TypeScript 是 Web 开发的必备语言，正好通过项目学习

> ⚠️ 你不需要事先学 JavaScript。整个项目由 Copilot 编写代码，你负责审核和决策。在过程中自然会建立对 Web 开发的理解。

## 四、数值体系设计

### 4.1 基础属性（D&D 六维）

| 属性 | 英文 | 对战作用 |
|------|------|----------|
| 力量 | STR | 物理攻击伤害 |
| 敏捷 | DEX | 闪避率、先手顺序 |
| 体质 | CON | 生命值上限、物理减伤 |
| 智力 | INT | 技能伤害加成 |
| 感知 | WIS | 暴击率、技能抵抗 |
| 魅力 | CHA | 特殊物品效果加成、对战评价 |

- 数值范围：3-20（对标 D&D）
- 总点数预算：约 60-80（由 LLM 在此范围内分配，保持平衡但有个性）

### 4.2 装备与技能

| 类型 | 数量上限 | 属性 |
|------|----------|------|
| 武器 | 3 | 名称、类型、攻击力、特殊效果 |
| 防具 | 3 | 名称、类型、防御力、特殊效果 |
| 技能 | 3 | 名称、来源、伤害/效果、冷却回合数 |
| 特殊物品 | 3 | 名称、描述、效果（可以是 0 效果的纯装饰） |

### 4.3 对战公式核心思路

```
每回合流程：
1. 先手判定：DEX 高者先行动（相同则随机）
2. 行动选择：普通攻击 / 使用技能（系统自动选择最优策略）
3. 命中判定：基础命中率 80% + (攻方DEX - 守方DEX) × 2%
4. 伤害计算：
   - 物理伤害 = 武器攻击力 + STR × 1.5 - 防具防御力 - CON × 0.5
   - 技能伤害 = 技能基础伤害 + INT × 2.0 - WIS × 0.8
5. 暴击判定：暴击率 = 5% + WIS × 1%
6. 特殊物品触发：每回合概率触发，概率和效果由物品属性决定
```

> ⚠️ 这是初始公式，需要通过实际对战测试后调优。可能出现的问题：
> - 数值碾压（高属性角色永远赢）
> - 技能过强/过弱
> - 某类角色构型 dominant
> 计划在 Phase 2 结束后做一轮模拟对战平衡性测试。

## 六、数据库设计（初版）

```sql
-- 用户表
users (
  id            UUID PRIMARY KEY,
  name          VARCHAR(8) NOT NULL,     -- 大写英文字母
  code_hash     VARCHAR(64) NOT NULL,    -- 4位识别码的哈希
  created_at    TIMESTAMP DEFAULT NOW(),
  UNIQUE(name, code_hash)                -- 名字+识别码唯一
)

-- 角色表
characters (
  id            UUID PRIMARY KEY,
  user_id       UUID REFERENCES users,   -- NULL = 系统角色
  name          VARCHAR(50) NOT NULL,     -- 角色名
  keywords      TEXT,                     -- 生成用的关键词
  description   TEXT,                     -- 角色描述
  -- D&D 六维
  str           INT NOT NULL,
  dex           INT NOT NULL,
  con           INT NOT NULL,
  int_val       INT NOT NULL,
  wis           INT NOT NULL,
  cha           INT NOT NULL,
  -- 装备和技能（JSON）
  weapons       JSONB,
  armors        JSONB,
  skills        JSONB,
  items         JSONB,
  -- 元数据
  is_system     BOOLEAN DEFAULT FALSE,
  created_at    TIMESTAMP DEFAULT NOW()
)

-- 对战记录表
battles (
  id            UUID PRIMARY KEY,
  player1_id    UUID REFERENCES characters,
  player2_id    UUID REFERENCES characters,
  winner_id     UUID REFERENCES characters,
  battle_log    JSONB,                   -- 完整对战日志
  summary       TEXT,                    -- 综合评价
  created_at    TIMESTAMP DEFAULT NOW()
)

-- 每日限额追踪表
daily_limits (
  id            UUID PRIMARY KEY,
  user_id       UUID REFERENCES users,
  date          DATE NOT NULL,
  generations   INT DEFAULT 0,           -- 当日生成次数
  battles       INT DEFAULT 0,           -- 当日对战次数
  UNIQUE(user_id, date)
)
```

## 七、风险评估

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| LLM 生成的角色数值不平衡 | 高 | 中 | 通过 Prompt 约束总点数范围 + 后端校验 |
| LLM API 偶尔返回格式异常 | 中 | 中 | 严格的 JSON Schema 校验 + 重试机制 |
| 对战公式不有趣/不平衡 | 中 | 高 | Phase 2 后做模拟对战测试，数据驱动调参 |
| Vercel 免费额度不够 | 低 | 中 | Hobby 计划支持 100GB 带宽/月，小规模足够 |
| 像素风视觉效果不理想 | 中 | 中 | 可参考现有像素风游戏（如 Noita, Celeste 配色方案） |
| 账号系统安全性（4位码太简单） | 中 | 低 | 一期接受，二期可升级为更安全的认证方式 |
